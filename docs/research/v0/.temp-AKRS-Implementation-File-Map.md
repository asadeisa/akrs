# AKRS Workflow — Implementation File Map (Report)

> **What this report is:** A complete list of every **file** I would create to implement
> an AKRS workflow on a real project, organized as a tree so the structure is clear.
>
> **What this report is NOT:** It does **not** contain any file *contents* — only the
> file **names**, their **location**, and one line explaining the **single purpose** of each
> (per the Constitution's "every file answers one purpose" rule).
>
> **Scope note:** Built only from the four source documents you gave me
> (`Constitution.md`, `Execution-Contract.md`, `Appendix to the AKRS Constitution.md`,
> `akrs-system-graph.drawio`). No other files were scanned.

---

## 0. Example project used for this map

AKRS is project-agnostic, so to make the tree concrete I assume an example
**e-commerce backend** as the confirmed Source of Truth. From it the Leader would
discover four **business capabilities** → four **Plans**:

| Plan | Business capability |
|------|---------------------|
| `auth`     | Identity, login, sessions |
| `catalog`  | Products, categories, search |
| `orders`   | Cart, checkout, order lifecycle |
| `payments` | Charges, refunds, webhooks |

The file *names* below are the only thing that changes between projects;
the *structure and responsibilities* stay identical (the "determinism" goal in the Appendix).

---

## 1. Top-level layout

The AKRS workflow lives in its **own folder** (`.akrs/`), kept strictly separate from
the project code (`src/`). Workers edit `src/`; they navigate `.akrs/`.

```
my-ecommerce-backend/
├── .akrs/                     ← the entire AKRS workflow (this report)
├── src/                       ← PROJECT CODE — Roads point here, Worker edits here
│                                 (NOT part of AKRS, NOT scanned freely)
└── docs/                      ← source-of-truth docs (requirements, architecture)
```

## 2. Full `.akrs/` file tree

Legend:
- **[A]** = built in **Phase A** (Initial Generation — the stable skeleton, built once)
- **[B]** = built in **Phase B** (On-Demand — one Task + one Road per request)
- **[S]** = **Spec/Schema** scaffolding (closes the "no file format" gap in the diagram)

```
.akrs/
│
├── akrs.manifest.md                         [A]  Root entry point: version, what each folder owns, build status
├── source-of-truth.md                       [A]  Confirmed authoritative sources (MANDATORY first artifact)
│
├── schemas/                                       Defines the SHAPE of every generated file (GAP A)
│   ├── router.schema.md                     [S]  Required fields/shape of a Router file
│   ├── memory.schema.md                     [S]  Required shape of a Memory index file
│   ├── road.schema.md                       [S]  Required shape of a Road (execution contract)
│   ├── task.schema.md                       [S]  Required shape of a Task
│   ├── plan.schema.md                       [S]  Required shape of a Plan
│   └── phase.schema.md                      [S]  Required shape of a Phase
│
├── router/                                        "Where should execution go next?" — routes only
│   ├── router.md                            [A]  Master router: prompt → Plan/Phase entry point
│   ├── mode-selection.md                    [A]  Prompt → Mode (0–4) decision table (GAP B)
│   └── routes/                                    One narrow route map per Plan
│       ├── auth.route.md                    [A]  Routes inside the auth capability
│       ├── catalog.route.md                 [A]  Routes inside the catalog capability
│       ├── orders.route.md                  [A]  Routes inside the orders capability
│       └── payments.route.md               [A]  Routes inside the payments capability
│
├── memory/                                        "Which knowledge?" — indexes, never documentation
│   ├── memory-index.md                      [A]  Top-level index of all memory files + owners
│   ├── capabilities/                              One memory per business capability
│   │   ├── auth.memory.md                   [A]  Index → where auth knowledge lives
│   │   ├── catalog.memory.md               [A]  Index → where catalog knowledge lives
│   │   ├── orders.memory.md                [A]  Index → where order knowledge lives
│   │   └── payments.memory.md              [A]  Index → where payment knowledge lives
│   ├── shared/                                    Reusable across ≥2 capabilities (the only reason to exist)
│   │   ├── database.memory.md               [A]  Owner-index for DB models/migrations
│   │   ├── api-conventions.memory.md        [A]  Owner-index for routing/validation conventions
│   │   ├── auth-context.memory.md           [A]  Owner-index for the shared auth/session primitive
│   │   └── error-handling.memory.md         [A]  Owner-index for shared error model
│   └── dependency.memory.md                 [A]  Impact analysis — relationships ONLY (what breaks what)
│
├── plans/                                         One Plan = one business capability
│   ├── auth.plan.md                         [A]  Auth capability: objectives, owned phases
│   ├── catalog.plan.md                      [A]  Catalog capability
│   ├── orders.plan.md                       [A]  Orders capability
│   └── payments.plan.md                     [A]  Payments capability
│
├── phases/                                        Milestones grouped under each Plan
│   ├── auth/
│   │   ├── auth-p1-foundation.phase.md      [A]  e.g. user model + registration milestone
│   │   └── auth-p2-sessions.phase.md        [A]  e.g. login/session/token milestone
│   ├── catalog/
│   │   ├── catalog-p1-products.phase.md     [A]
│   │   └── catalog-p2-search.phase.md       [A]
│   ├── orders/
│   │   ├── orders-p1-cart.phase.md          [A]
│   │   └── orders-p2-checkout.phase.md      [A]
│   └── payments/
│       ├── payments-p1-charges.phase.md     [A]
│       └── payments-p2-webhooks.phase.md    [A]
│
├── tasks/                                         On-demand ONLY — empty at skeleton time
│   └── orders/
│       └── orders-p2-t1-create-order.task.md [B] Example: generated when "create order" is requested
│
├── roads/                                         On-demand ONLY — exactly ONE Road per Task
│   └── orders/
│       └── orders-p2-t1-create-order.road.md [B] Execution contract for the Task above
│
└── validation/                                    Defines WHAT/ WHO validates (GAP F)
    ├── skeleton-validation.md               [A]  Phase-A checklist (one owner, no dup, router=routes…)
    ├── task-road-validation.md              [A]  Phase-B checklist run before hand-off to Worker
    └── workflow-checklist.md                [A]  Master "is the workflow complete?" gate
```

## 3. Phase A vs Phase B — what exists *when*

The Appendix forbids merging the two generation phases, so the tree fills in two waves:

### Phase A — Initial Generation (built once, the skeleton)
Everything marked **[A]** and **[S]** above. Critically, Phase A produces **NO** tasks and **NO** roads:

```
.akrs/
├── akrs.manifest.md
├── source-of-truth.md
├── schemas/*            (all)
├── router/*             (all)
├── memory/*             (all)
├── plans/*              (all)
├── phases/*             (all)
└── validation/*         (all)

tasks/   →  (empty)
roads/   →  (empty)
```

### Phase B — On-Demand Generation (one request → one Task + one Road)
Each request adds **exactly two** files (a `.task.md` and its single `.road.md`),
then they are validated and handed to the Worker:

```
tasks/orders/orders-p2-t1-create-order.task.md   +
roads/orders/orders-p2-t1-create-order.road.md
```

Repeat per request — never in advance, never mass-generated.

## 4. How each layer answers exactly one question

Mapping the tree back to the Constitution's "each layer = one question" rule
(and the diagram's runtime navigation lane):

```
Prompt
  │
  ▼
router/mode-selection.md ──►  Which Mode (0–4)?
  │
  ▼
router/router.md + router/routes/*  ──►  Where next?
  │
  ▼
memory/*  ──►  Which knowledge?
  │
  ▼
roads/<…>.road.md  ──►  Exactly what to read? (points into src/)
  │
  ▼
tasks/<…>.task.md  ──►  Exactly what to build?
  │
  ▼
src/  ──►  Worker executes / edits here
```

## 5. Ownership summary (one concept → one file)

| Concept | Single owner file |
|---------|-------------------|
| Authoritative sources | `.akrs/source-of-truth.md` |
| Prompt → Mode mapping | `.akrs/router/mode-selection.md` |
| Global routing | `.akrs/router/router.md` |
| What knowledge exists & where | `.akrs/memory/memory-index.md` + `memory/**` |
| Cross-file impact | `.akrs/memory/dependency.memory.md` |
| A business capability | `.akrs/plans/<cap>.plan.md` |
| A milestone of work | `.akrs/phases/<cap>/<…>.phase.md` |
| What to build (one objective) | `.akrs/tasks/**/<…>.task.md` |
| What to read / change scope | `.akrs/roads/**/<…>.road.md` |
| File shapes (determinism) | `.akrs/schemas/*` |
| Completion gates | `.akrs/validation/*` |

---

## 6. Counts at a glance (for this 4-capability example)

- **Phase A skeleton files:** ~38 files
  (1 manifest + 1 source-of-truth + 6 schemas + 6 router + 11 memory + 4 plans + 8 phases + 3 validation)
- **Phase B files:** **2 per request** (1 task + 1 road), created on demand — `tasks/` and `roads/` start empty.

> Renaming any file is allowed across projects; the **folders, responsibilities, and the
> one-owner / routes-only / one-road-per-task rules are fixed**.
