# Versioning

AKRS has **three independent version lines**. Keeping them separate is what lets
the framework evolve without forcing migrations on existing projects.

| Version line | What it versions | Where it lives | Who bumps it |
|--------------|------------------|----------------|--------------|
| **Framework Version** | The specification in `docs/framework/` | This repo / npm package | Maintainers |
| **Workflow Version** | A generated workflow inside a target project | The target project | The Leader, per project |
| **Kernel Version** | The compiled `KERNEL.md` for one project | The target project | Regenerated per project |

---

## Framework Version (this package)

Follows **Semantic Versioning** — `MAJOR.MINOR.PATCH`.

- **MAJOR** — a change that breaks how existing workflows are generated or
  executed (e.g. removing an artifact layer, changing the route).
- **MINOR** — new, backward-compatible capability (e.g. a new optional spec, a
  new adapter, a CLI command).
- **PATCH** — clarifications, typo fixes, non-behavioral doc improvements.

The current framework version is the `version` field in `package.json` and the
top entry of [`CHANGELOG.md`](CHANGELOG.md).

> An architectural principle in `01-Constitution.md` changing is **always** a
> MAJOR. Wording that clarifies an existing principle is a PATCH.

---

## Workflow Version (per project)

Each generated project may record which framework version produced it, e.g. in
`akrs/STATE.md` or a header in `KERNEL.md`:

```
Generated with AKRS framework v1.0.0
```

This is informational. A project does **not** have to upgrade when the framework
does — its Kernel keeps working.

---

## Kernel Version (per project)

The Kernel is a **compiled artifact**, not a versioned source file:

- It is **regenerated**, never hand-migrated.
- It is **never** stored in this framework repo.
- It is **never** reused across projects.

To "upgrade" a project's Kernel, regenerate it from the latest framework with the
strongest available model. Because models improve over time, a regenerated
Kernel is generally *better* than the one it replaces.

---

## Compatibility Promise

- Within a MAJOR line, a workflow generated by an earlier MINOR keeps executing.
- Upgrading the framework never requires editing a project's generated files —
  at most, you regenerate the Kernel.
- Research and v0 material under `docs/research/` is frozen and never changes
  behavior.
